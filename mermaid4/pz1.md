# Практическое задание

**Sequence Diagram в Mermaid**

## Цель

Закрепить навыки **моделирования поведения системы во времени** с использованием:

* акторов и участников,
* различных типов стрелок,
* управляющих блоков `alt`, `loop`, `opt`,
* корректного разделения ответственности,
* читаемой и логически непротиворечивой UML sequence diagram.

После выполнения задания вы должны уметь:

* выбрать правильный тип стрелки по смыслу,
* определить границу системы,
* показать альтернативные сценарии,
* избежать типичных ошибок моделирования.

## Задание

### Сценарий

Смоделировать **процесс авторизации пользователя в веб-приложении**.

Система включает:

* Пользователя (внешний актор),
* Веб-клиент,
* Backend-сервис авторизации,
* Базу данных пользователей,
* Сервис уведомлений.

### Обязательные условия

В диаграмме **обязательно должны быть**:

1. **Актор**

   * Пользователь объявлен как `actor`.

2. **Участники системы**

   * Веб-клиент
   * Backend-сервис
   * База данных
   * Сервис уведомлений

3. **Типы стрелок**

   * Синхронный вызов (`->>`)
   * Ответ (`-->>`)
   * Асинхронное сообщение (`-)`)

4. **Управляющие блоки**

   * `alt` — успешная и неуспешная авторизация
   * `opt` — отправка уведомления при ошибке
   * `loop` — повторная попытка ввода пароля (ограниченное число раз)

5. **Логика сценария**

   * Пользователь инициирует процесс
   * Backend проверяет данные в базе
   * В зависимости от результата:

     * либо происходит успешный вход,
     * либо возвращается ошибка
   * При ошибке возможна повторная попытка
   * При критической ошибке отправляется уведомление

## Задание (шаги)

1. Опишите сценарий **словами** (5–7 предложений).
2. Определите:

   * кто является актором,
   * какие участники относятся к системе.
3. Решите:

   * где используется синхронный вызов,
   * где асинхронное сообщение.
4. Определите:

   * где нужен `alt`,
   * где `opt`,
   * где `loop`.
5. Напишите **код Mermaid sequence diagram**, соблюдая UML-семантику.
6. Проверьте читаемость диаграммы:

   * логичный порядок,
   * минимальная вложенность,
   * осмысленные названия сообщений.

## Подсказки по ключевым частям

### Актор

* Пользователь **не должен** напрямую взаимодействовать с базой данных.
* Все действия пользователя идут через клиент.

### Стрелки

* Вызовы проверки данных — **синхронные**.
* Ответы базы данных — **пунктирные**.
* Уведомления — **асинхронные**, без ожидания ответа.

### `alt`

* Ветви должны быть **взаимоисключающими**:

  * `[валидные данные]`
  * `[ошибка авторизации]`

### `loop`

* Укажите условие:

  * количество попыток,
  * или логическое ограничение.

### `opt`

* Используйте только там, где действие **необязательное**.

## Что проверить перед отправкой (чек-лист)

Перед сдачей работы убедитесь, что:

* [ ] Есть **ровно один инициатор сценария**
* [ ] Пользователь объявлен через `actor`
* [ ] Внутренние компоненты — `participant`
* [ ] Все стрелки выбраны **по смыслу**, а не «для красоты»
* [ ] `alt` содержит **логические условия**
* [ ] `loop` не используется без необходимости
* [ ] Нет избыточных ответных стрелок
* [ ] Диаграмма описывает **один сценарий**, а не всё приложение
* [ ] Названия сообщений — глаголы, а не существительные

## Советы по улучшению работы

1. **Не пишите код сразу**
   Сначала продумайте сценарий логически.

2. **Избегайте архитектурного мышления**
   Sequence diagram — про поведение, а не про устройство системы.

3. **Минимизируйте количество участников**
   Если участник ничего не делает — он лишний.

4. **Не бойтесь удалять элементы**
   Хорошая диаграмма чаще получается после упрощения.

5. **Проверяйте диаграмму вопросом**:

   > «Смогу ли я по ней объяснить процесс другому человеку за 2 минуты?»

Если ответ «нет» — диаграмму нужно переработать.

### Результат выполнения

В итоге у вас должен быть:

* один файл с Mermaid-кодом,
* одна sequence diagram,
* чётко читаемый сценарий авторизации,
* корректное использование UML-семантики.
