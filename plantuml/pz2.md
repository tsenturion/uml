## Практическое задание: моделирование архитектуры и взаимодействий в PlantUML

### Цель

Закрепить понимание различий между уровнями архитектурного моделирования: логическая структура системы, взаимодействие компонентов и физическое развертывание. Научиться выбирать корректный тип диаграммы и использовать базовые элементы PlantUML: use case, component и deployment. Дополнительно — отработать принцип разделения ответственности между слоями системы и корректное отображение зависимостей.

## Задание (шаги)

Вам необходимо смоделировать интернет-сервис (например, систему онлайн-бронирования или интернет-магазин) и создать три диаграммы в PlantUML.

### Шаг 1. Use Case диаграмма (требования)

Создайте диаграмму, которая описывает взаимодействие внешних акторов с системой.

Обязательные элементы:

- минимум 2 актора (например: пользователь, администратор)
- минимум 4 use case
- хотя бы одно отношение `<<include>>`
- хотя бы одно отношение `<<extend>>`
- граница системы (rectangle)

Сценарий должен включать:

- вход в систему
- основное действие пользователя (например, заказ/заявка)
- административный сценарий (например, просмотр отчётов)

### Шаг 2. Component диаграмма (архитектура системы)

Смоделируйте логическую архитектуру приложения.

Обязательные элементы:

- минимум 4 компонента (UI, сервисы, БД, внешний сервис)
- минимум 3 интерфейса
- зависимости компонентов через интерфейсы
- хотя бы один внешний сервис

Требуется показать:

- кто предоставляет интерфейс
- кто его использует
- отсутствие прямых зависимостей «компонент -> компонент»

### Шаг 3. Deployment диаграмма (инфраструктура)

Смоделируйте физическое развертывание системы.

Обязательные элементы:

- минимум 3 узла (client, server, database)
- размещение артефактов на каждом узле
- сетевые связи между узлами
- подписи протоколов (HTTP/HTTPS, SQL или TCP/IP)

## Подсказки по ключевым частям

Use Case:

- акторы всегда находятся вне системы
- use case описывают «что делает система», а не «как»
- include = обязательное поведение
- extend = условное поведение

Component:

- интерфейсы — это контракты, а не реализация
- зависимости должны идти к интерфейсам, а не к компонентам
- компонент = модуль системы, а не класс

Deployment:

- node = физический или виртуальный ресурс
- artifact = то, что развернуто на узле
- стрелки = сетевые каналы, а не вызовы методов

## Что проверить перед отправкой (чек-лист)

- [ ] Use Case диаграмма содержит акторов вне системы
- [ ] Use Case не содержит описания алгоритмов выполнения
- [ ] В Component диаграмме используются интерфейсы, а не прямые связи компонентов
- [ ] Каждый компонент имеет хотя бы один интерфейс (предоставляемый или используемый)
- [ ] В Deployment диаграмме есть минимум 3 узла
- [ ] Артефакты размещены внутри узлов, а не отдельно
- [ ] Все связи в Deployment подписаны протоколами
- [ ] Нет смешивания уровней (например, логики процесса в Deployment)

## Советы по улучшению работы

Если диаграмма получается перегруженной, значит вы смешали уровни абстракции. Разделяйте:

- требования -> Use Case
- архитектура -> Component
- инфраструктура -> Deployment

Старайтесь избегать ситуаций, где один и тот же элемент «повторяется» на разных диаграммах без изменения смысла. Например, «User» в Use Case не равен «User Service» в Component и не равен «Client Device» в Deployment.

Для усложнения задания можно добавить:

- балансировщик нагрузки в Deployment
- сервис уведомлений в Component
- альтернативные сценарии авторизации в Use Case

Главный критерий качества — не количество элементов, а отсутствие логических противоречий между уровнями модели.
