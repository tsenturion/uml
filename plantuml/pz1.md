### Практическое задание: Use Case Diagram в PlantUML

### Цель

Научиться формально описывать требования к системе через use case диаграмму: корректно выделять акторов, границу системы, варианты использования и типы связей (`association`, `<<include>>`, `<<extend>>`, обобщение).

### Задание (шаги)

Необходимо реализовать use case диаграмму для системы **онлайн-бронирования билетов (кино/театр)**.

1. Описать границу системы с названием **«Система бронирования»**.

2. Внутри системы определить use cases:
   - «Зарегистрироваться»
   - «Войти в систему»
   - «Выбрать сеанс»
   - «Выбрать места»
   - «Оплатить билет»
   - «Применить промокод»
   - «Получить электронный билет»
   - «Просмотреть отчёт по продажам»

3. Добавить акторов:
   - «Пользователь»
   - «Администратор»

4. Задать обобщение акторов:
   - Администратор — частный случай Пользователя

5. Связать акторов с use cases:
   - Пользователь взаимодействует с процессом покупки билета (вход, выбор, оплата)
   - Администратор — только с отчётами

6. Добавить связи `<<include>>`:
   - «Оплатить билет» всегда включает «Выбрать места»
   - «Выбрать места» всегда включает «Выбрать сеанс»

7. Добавить связь `<<extend>>`:
   - «Применить промокод» расширяет «Оплатить билет»

8. Завершить диаграмму корректным синтаксисом (`@startuml` / `@enduml`).

### Подсказки по ключевым частям

Граница системы — это не декоративный элемент, а строгое разделение ответственности. Всё, что внутри, — функции системы. Акторы всегда находятся снаружи.

Use case — это **ценность для актора**, а не технический шаг.
Например, «Оплатить билет» — корректно, а «Отправить SQL-запрос» — нет.

Связь `<<include>>` используется, когда действие **всегда происходит** как часть другого. Это декомпозиция.
Связь `<<extend>>` — когда поведение **добавляется условно**.

Обобщение акторов (`--|>`) означает наследование всех сценариев. Не дублируйте связи вручную, если используете наследование.

### Что проверить перед отправкой (чек-лист)

- Все акторы находятся **вне** границы системы
- Все use cases находятся **внутри** системы
- Нет смешения уровней (нет технических операций вместо пользовательских сценариев)
- `<<include>>` направлен **от базового сценария к включаемому**
- `<<extend>>` направлен **от расширения к базовому сценарию**
- Обобщение акторов задано корректно (`--|>`)
- Нет дублирующих или избыточных связей
- Названия use case — в форме действия (глагол + объект)
- Диаграмма компилируется без ошибок

### Советы по улучшению работы

Старайтесь минимизировать количество use cases. Если их слишком много, модель теряет читаемость. Лучше выделять только ключевые сценарии.

Не используйте диаграмму как алгоритм. Если появляется желание показать порядок шагов — это уже задача для activity или sequence диаграммы.

Следите за уровнем абстракции. Все use cases должны быть примерно одного масштаба. Если один описывает бизнес-процесс, а другой — мелкий технический шаг, это признак неправильной декомпозиции.

Используйте `<<include>>` для устранения дублирования. Если один и тот же шаг повторяется в нескольких сценариях, его стоит вынести в отдельный use case.

Добавляйте `<<extend>>` только там, где действительно есть условность. Избыточное использование этой связи усложняет модель и ухудшает её интерпретацию.

Визуальная аккуратность тоже имеет значение. Даже при корректной модели перегруженная диаграмма становится трудной для восприятия.
